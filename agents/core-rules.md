# 核心开发红线 (Always-On Rules)

以下规则在每个会话中始终生效，无需额外加载。

## Spec 驱动

1. **先 Spec 再代码**：任何代码修改前，必须先更新对应 SPEC.md。严禁未更新 Spec 就偷跑功能。

## 工程质量红线

2. **治本原则（五问法）**：遇到任何问题时，必须连续追问"为什么"至少 5 层，直到触及架构或定位层面的根因，再从根因修复。禁止在链路末端掩盖症状。此原则适用于所有层面——代码 bug、项目配置、平台决策，无一例外。
3. **KISS 约束**：修改一行配置优于增加一段脚本。复杂度是错误的避难所。
4. **闭环验证**：成功必须端到端验证，仅凭进程状态判定成功视为失职。
5. **禁止补丁叠补丁**：补丁导致二次故障时，必须撤销所有临时逻辑并重新根源分析。

## opencode 原生优先

6. **opencode 原生机制优先**：实现任何功能或约束行为前，必须先查阅 https://opencode.ai/docs/ 确认 opencode 是否原生支持。只有原生不支持时，才考虑其他方案。

## 替换清理

7. **替换清理准则**：引入新机制替代旧机制时，必须清理旧机制中被完全替代的部分。只加不减会导致同一约束被多个机制双重声明，造成冗余和潜在冲突。判断标准：如果新机制能达到旧机制同样的效果，旧机制中对应的部分应删除，仅保留新机制未覆盖的部分。

## 开发原则

8. **迭代追溯**：需求 → REQUESTS.md → SPEC.md → 代码，严格按序执行。
